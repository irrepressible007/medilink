================================================================================
  SECTION 1 — PLATFORM OVERVIEW
================================================================================

MediLink is a full-stack, mobile-first healthcare discovery and management
platform built specifically for Bangladesh. It serves three user roles —
Patients, Doctors, and Admins — within a single unified web application.

The platform allows patients to discover hospitals and medical services,
book appointments, chat with doctors in real time, perform AI-assisted
symptom triage, share medical records, and dispatch GPS-based SOS emergency
alerts. Doctors receive, manage, and respond to those interactions through a
dedicated clinical dashboard. Admins monitor the entire platform through a
command center featuring live analytics and an emergency dispatch map.

  Live URL   : https://medilink-ebon.vercel.app
  Repository  : https://github.com/irrepressible007/medilink
  Frontend    : Vercel (CDN-distributed, auto-deploy on push to main)
  Backend     : Render (Node.js persistent web service)
  Database    : MongoDB Atlas (cloud-hosted NoSQL)

================================================================================
  SECTION 3 — SYSTEM DESIGN
================================================================================

3.1  Authentication Design
---------------------------
  - Three separate auth flows: Patient, Doctor, Admin.
  - All flows use the same /api/auth router with role-specific endpoints.
  - Passwords are hashed with bcrypt (salt rounds: 10) before storage.
  - On successful login, the server signs a JWT containing:
      { userId, email, role }     expires in 7 days
  - The JWT is stored in localStorage keyed by role:
      medilink_token          → Patient
      medilink_doctor_token   → Doctor
      medilink_admin_token    → Admin
  - All protected API routes pass the token in the Authorization header:
      Authorization: Bearer <JWT>
  - The authMiddleware.js decodes and verifies the token, attaches
    req.user = { userId, email, role } to every request.
  - Socket.io also applies JWT auth via handshake middleware; invalid
    tokens are rejected before any event is established.

3.2  Rate Limiting Layers
--------------------------
  - Auth endpoints (login routes): max 20 requests per 15 minutes.
  - AI endpoint (/api/ai): max 15 requests per minute.
  - This prevents brute-force attacks and AI API cost overruns.

3.3  File Upload Pipeline
--------------------------
  Multer buffers the file in memory → Cloudinary SDK uploads it → 
  Cloudinary returns a secure_url → URL stored in MongoDB.
  Used in: Medical Records (lab results, X-rays), Messages (attachments).

3.4  Email Notification Pipeline
----------------------------------
  - SendGrid is the primary transactional email provider.
  - nodemailer is the secondary fallback (SMTP).
  - Emails are triggered:
      • Appointment created → confirmation to patient
      • Appointment status changed → update to patient
      • Prescription written → PDF-style details to patient
      • Follow-up approved → confirmation to patient
      • Follow-up requested → notification to doctor

3.5  Scheduled Tasks (node-cron)
----------------------------------
  A scheduler service runs inside the Node.js process on boot.
  Cron Job: Runs daily at 08:00 Bangladesh time.
  Logic: Queries all appointments scheduled for tomorrow with
         reminderSent=false, sends a reminder email to the patient,
         and updates reminderSent=true to prevent duplicate sends.

3.6  CORS Policy
-----------------
  Origins whitelisted:
    - http://localhost:5173  (Vite dev server)
    - http://localhost:3000  (alternate dev port)
    - process.env.CLIENT_URL (Vercel production URL, set on Render)
  Requests from any other origin are blocked with a 403-equivalent.

3.7  MongoDB Data Access Pattern
----------------------------------
  - Prisma Client is instantiated per route file (one per router module).
  - All queries use async/await with try/catch.
  - ObjectId fields use @db.ObjectId in the schema.
  - Relations are resolved using Prisma's include:{} syntax.
  - No in-memory caching layer (Redis) is used currently.

3.8  Frontend State Management
--------------------------------
  - No Redux / Zustand. All state is managed with React hooks.
  - Auth state: localStorage (token + serialised user JSON).
  - Component-level state: useState, useEffect, useRef, useCallback.
  - Real-time messages: maintained in a local messages array updated
    via Socket.io event listeners within useEffect.

3.9  Mobile-First Responsive Design
--------------------------------------
  - All pages designed mobile-first (375px base).
  - BottomNav component replaces sidebar navigation on mobile.
  - CSS breakpoints at 600px and 900px scale up to tablet/desktop.
  - Table overflows are horizontally scrollable; inputs are touch-sized.

  Authentication
  ──────────────
  Every socket connection must provide a valid JWT in handshake.auth.token.
  The JWT is decoded and verified server-side. Invalid tokens are rejected
  before any events can fire. Authenticated users automatically join a
  private socket room keyed by their userId.

  Events Implemented
  ───────────────────
  Client → Server:
    send_message     { receiverId, messageData }
                     Server relays messageData to the target user's room.

    call_user        { userToCall, signalData, from, name }
                     Server forwards a WebRTC SDP offer to target user.

    answer_call      { to, signal }
                     Server forwards a WebRTC SDP answer to the caller.

    ice_candidate    { to, candidate }
                     Server forwards an ICE candidate for WebRTC negotiation.

  Server → Client:
    receive_message  messageData
                     Emitted to the recipient's room when a message is sent.

    call_user        { signal, from, name }
                     Emitted to the callee when an inbound call arrives.

    call_accepted    signal
                     Emitted to the caller when the callee answers.

    ice_candidate    candidate
                     Emitted to both sides during WebRTC negotiation.

  Telemedicine Call Flow
  ───────────────────────
  1. Caller emits 'call_user' with their WebRTC SDP offer.
  2. Server relays it to the callee's socket room.
  3. Callee accepts → emits 'answer_call' with SDP answer.
  4. Server relays answer back to caller.
  5. Both sides exchange ICE candidates via 'ice_candidate' events.
  6. WebRTC peer-to-peer connection established → HD audio/video.

  Real-Time Messaging Flow
  ─────────────────────────
  1. Sender POSTs message to /api/messages/:conversationId (REST).
  2. Server saves message to MongoDB.
  3. Server also emits 'receive_message' to the recipient's socket room.
  4. Recipient's client receives the event and prepends message to UI.
     → Message appears instantly without any page refresh.

    Accept button (green) → PATCH /api/appointments/:id/status { status: "confirmed" }.
    Decline button (red) → PATCH with { status: "cancelled" }.
    Notes field inline → debounced PATCH to add clinical notes.

  TAB 2 — Patient Records:
    Search input for patient name → GET /api/profile/patients/:id.
    Selected patient's full medical record list displayed below.
    "Upload Record" section:
      Inputs: title, type dropdown, description, appointment link (optional), file.
      Submit → POST /api/records (multipart/form-data).
    "Write Prescription" section:
      Inputs: medication name, dosage, frequency, start date, end date.
      Submit → POST /api/prescriptions.
      Success triggers automatic email to patient.
    "Refer Patient" section:
      Doctor search for target specialist → POST /api/referrals.

  TAB 3 — Prescriptions:
    GET /api/prescriptions/doctor.
    Table: medication, dosage, frequency, start/end dates, active badge.

  TAB 4 — Messages:
    Identical to patient-facing Messages page.
    Fetches GET /api/messages/conversations.
    Real-time Socket.io message delivery.

  TAB 5 — Availability:
    7 rows (one per day of week).
    Each row: day name, "Off" checkbox, start time input, end time input.
    If "Off" checked: time inputs hidden; row greyed out.
    "Save Schedule" → PATCH /api/profile/doctor/me { workingHours: JSON }.
    Saved JSON is shown in patients' appointment booking flow.

10.16  Notification & Email System
──────────────────────────────────────────────────────────────────────────────
  Every significant patient-facing action triggers an email:
  
    Action                      | Email Sent To
    ────────────────────────────────────────────────────────
    Appointment created         | Patient (confirmation)
    Appointment status changed  | Patient (status update)
    Prescription written        | Patient (medication details)
    Follow-up requested         | Doctor (notification)
    Follow-up approved          | Patient (approval confirmation)
    Tomorrow's appointment      | Patient (reminder, via cron job at 08:00)
  
  All notification records are written to the Notification collection
  including: type, channel, subject, body, success boolean, and errorMsg
  if the send failed. This provides a full audit trail.

10.17  Deployment & CI/CD Pipeline
──────────────────────────────────────────────────────────────────────────────
  1. Developer pushes code to GitHub (main branch).
  2. Vercel detects the push, runs "npm run build" in /client.
  3. Vite compiles React → static assets in /client/dist.
  4. Vercel deploys the static files to the global CDN.
  5. Render detects the push, rebuilds the Node.js server in /server.
  6. Render restarts the persistent web service.
  7. New API routes and frontend code are live within ~1-2 minutes.
  
  Schema Changes: If prisma/schema.prisma is modified, "npx prisma db push"
  must be run manually from the server directory against the production
  MongoDB Atlas URI to sync collection/index changes.
